    TLDR

    Upgraded and consolidated a Lenovo ThinkCentre M720s into a private cloud homelab running Proxmox VE. The system now features 32GB RAM, ZFS-backed primary storage, automated snapshots, and four streamlined VMs handling reverse proxy (screened subnet), media streaming, security tooling, and infrastructure management. All Docker workloads route through Gluetun for outbound isolation. The environment integrates with a broader Zero Trust network architecture via VLAN-aware bridging, MAC security, and Cloudflare tunnels. Regular backups and infrastructure snapshots provide strong recoverability. The homelab balances performance, security, and maintainability while significantly reducing previous sprawl.

    Introduction

    What began as a basic media server has evolved into a consolidated, secure, and resilient private cloud homelab. Running on Proxmox with strict network segmentation, ZFS storage, automated backups, and Docker-based services, the environment delivers full self-hosted media streaming and infrastructure management with no recurring service fees.

    Hardware & Operating Systems

    Beyond the base Lenovo M720s chassis, the following hardware configuration was implemented to support the Proxmox homelab environment.


    Chassis: Small Form Factor design limits internal expansion options, particularly for full-height PCIe cards or additional drives. Cooling and low-profile component availability remain considerations for future upgrades.


    Central Processing Unit (CPU): Intel Core i5-8400 (6 cores, 2.8 GHz base). The integrated Intel Quick Sync hardware transcoder continues to provide efficient 4K media transcoding without requiring a discrete GPU.


    Random Access Memory (RAM): Upgraded from the original 24GB configuration to a total of 32GB DDR4. This provides ample headroom for consolidated workloads and concurrent service operation.


    Storage (Boot): 512GB NVMe SSD installed in the factory M.2 slot with an aftermarket caddy and custom heatsink for thermal management and longevity.


    Storage (Primary): Single 8TB Seagate IronWolf HDD configured as a ZFS pool named tank. While a single-disk ZFS setup does not provide redundancy, it delivers data integrity through checksums and self-healing capabilities.


    Storage (Backup): 8TB external desktop HDD (external1) connected via USB 3.0. Serves as the target for weekly backups of the tank pool and VM/container snapshots.


    Network Interface: Integrated Intel I219-V Gigabit NIC. The Proxmox host bridge has been reconfigured as VLAN-aware to support tagged traffic for the segmented network.


    This hardware foundation balances cost, performance, and reliability for the current homelab workload.

    Operating Systems & Virtualization

    Proxmox VE runs as the bare-metal hypervisor, providing centralized web-based management of all virtual machines and storage. Built on Debian, it offers robust ZFS support, snapshotting, and networking flexibility.


    All workloads now run as Ubuntu Server virtual machines. Docker Compose manages application stacks inside these VMs, ensuring isolation and avoiding nesting complications. This approach prioritizes security and operational simplicity over the lighter resource footprint of LXCs.

    Storage & Backup

    The storage and backup strategy was designed for data integrity, snapshot-based recovery, and secure outbound traffic.


    ZFS Pool: Primary storage uses a ZFS pool named tank on the 8TB Seagate IronWolf HDD. It provides checksums for data integrity and supports efficient snapshots.


    Backup Schedule: Weekly full backups of the tank pool are performed to the 8TB external USB drive (external1).


    Automated Snapshots: VM and container snapshots are automatically taken every Sunday to enable quick rollback if needed.


    This setup ensures reliable data protection while maintaining operational security for containerized services.

    Networking Integration

    The homelab is fully integrated with the existing Zero Trust-inspired network architecture. The Proxmox host connects to the UniFi switch via a trunk port with MAC address whitelisting. VLAN-aware bridging allows each VM to operate on its designated segment.


    Access VM: Resides in the tightly restricted /30 screened subnet (DMZ).


    VLAN Placement: Other VMs are placed according to their risk and function profiles across the segmented network.


    Security Benefit: This design limits lateral movement. Even if the DMZ is breached, the minimal IP space and strict firewall controls provide time to detect and respond using previous snapshots.


    This integration ensures the Proxmox environment adheres to the same Zero Trust principles as the rest of the infrastructure.

    Security & Hardening

    Security follows defense-in-depth principles aligned with Zero Trust tenets.


    SSH Hardening: Root SSH login disabled across all systems with key-based authentication and high-entropy passphrases enforced. Helps prevent local machines from becoming an attack surface. Passphrases are stored non-local, offline.


    Docker Outbound Protection: All Docker stacks route through Gluetun for outbound traffic encryption. DoT enabled.


    WAF (Cloudflare): Placed in front of all applications as a compensating control; helps with rate limiting, DOS attacks, and restricts common exploit paths. Also proxies my public IP address.


    Network Controls: VLAN segmentation, MAC port security, BPDU Guard, and strict OPNsense firewall rules provide the primary protection boundary.


    Recovery Capability: Infrastructure snapshots enable quick recovery from potential compromise. I will convert to a high availability structure as soon as I can.


    Future Enhancements: Planned additions include a RADIUS server for enhanced authentication and Proxmox-level firewall rules as an additional defense layer.


    This layered approach significantly reduces the attack surface and supports rapid recovery if needed.

    Challenges & Lessons Learned

    Consolidation introduced several operational and technical challenges that strengthened overall practices:


    LXC vs VM Compatibility: Mixing LXC containers with VMs led to persistent network and configuration issues. LXCs also presented Docker nesting quirks and lower security assurances, leading to a VM-centric standardization.


    Gluetun Docker Race Condition: Post-consolidation, timing issues arose when starting Gluetun alongside dependent containers. Explicit startup ordering (Gluetun first) was implemented to mitigate the issue, however - completely manual.


    ZFS Single-Disk Trade-offs: Self-healing and checksums provide strong integrity, but the lack of redundancy increases dependence on regular backups and snapshots.


    Performance Tuning After Consolidation: Resource allocation (CPU and RAM shares) required measurement and adjustment to optimize denser VM workloads. Originally, I had issued extra CPU, RAM, and storage - but I was wasting resources. Online specs don't necessarily translate to direct allocations.


    Dependency Mapping: NFS mounts and service startup order across reboots and network events demanded careful planning and testing.


    Physical Security: The stack continues to operate without a fully locked rack or enclosure.


    Backup Window Management: Weekly full backups of the 8TB pool are time-intensive, requiring scheduling trade-offs.


    Documentation Overhead: Maintaining accurate VLAN mappings, service dependencies, and network diagrams remains an ongoing effort as the environment evolves.


    This layered approach significantly reduces the attack surface and supports rapid recovery if needed.
