LazyAdmin

    ← Back to Library

    TLDR

    Performed nmap recon and web enumeration with Gobuster, discovered a SweetRice CMS instance and extracted an admin hash from a SQL backup file, cracked it with Hashcat to log into the admin panel, exploited the Ads feature to inject and execute PHP code for command execution as www-data, found user.txt, and escalated to root by overwriting a script executed with sudo privileges.

    Challenge Description

    LazyAdmin is an easy-rated Linux machine on TryHackMe featuring a web server running an older version of the SweetRice content management system, along with SSH access. The challenge requires compromising the web application to gain initial access to the system, followed by privilege escalation on the Linux host to reach root.

    Reconnaissance

    As always, I started by running NMAP, followed by searchsploit for immediate version vulnerabilities.

    I found two open ports: tcp/22 running OpenSSH 7.2p2 & tcp/80 running Apache 2.4.18. Since this version of Apache wasn't up to date, I ran searchsploit -w apache 2.4.18, which came back with an exploit involving local privilege escalation via the logrotate system.

    After further research, I found this exploit wouldn't serve me well in the immediate future, although a good vulnerability to fall back on. The exploit involves injecting code, then waiting for logrotate to automatically run (which takes place at 6:25 AM), which should escalate the user to root.

    I pulled the server up in-browser, but was met with the Default Apache Ubuntu page. Time to search for hidden directories.

    Enumeration

    First, I ran Gobuster against the root directory, and came back with one hit: /content

    If IP addresses change, ignore. This lab machine was having trouble staying online. I had to restart the instance several times.

    The /content page fetched an under construction template, name dropping something called 'SweetRice.'

    I did a quick google, turns out SweetRice is an open-source Content Management System (what 'CMS' stands for). The search literally told me SweetRice had significant security issues in the past, so I decided to run it through SearchSploit.



    Came back with quite a few hits, but the current running version of SweetRice is still in question so I moved to running gobuster on /content to hopefully come back with more directories.

    Gobuster returned multiple hidden directories & files for /content. Immediately I noticed changelog.txt - which after a quick inspection confirmed the latest version of SweetRice was, in fact, 1.5.1. The same version that came back with multiple vulnerabilities.

    I also found an admin login portal at /content/as - eluding that we'll need to search for credentials or brute-force them.

    After digging through the rest of the pathways, I also found a SQL backup file - which I quickly downloaded and searched.

    I searched the file and found a hash digest and a username manager, So I ran hashcat --identify, which brought back MD4 & MD5 as the highest likelihood. After running both algorithms against rockyou.txt - I cracked the password.

    Remember: hashcat --identify isn't a determinate answer, it's the most likely answer from top down.

    With the credentials in-hand, I logged right in and was brought to a website management dashboard.

    I cruised around the site, and did some light reading on SweetRice, but nothing stuck out to me. I revisited the SweetRice exploits from our earlier SearchSploit results, two stuck out to me.

    First exploit I tried was the Arbitrary File Download (#40698): The exploit is straightforward. You download the file and run it against your target.

    I made a few quality-of-life improvements, and adjusted the target URL.

    Notice: I had some trouble getting the exploit to work, because I added the '!' to 'Password123' by mistake. Even in the lowest reaches of my subconscious, I know Password123 is too weak. Come on now.

    It actually worked quite well, after I fixed some of the kinks. The only problem was that it didn't give much information about the current working directory, nor could I run commands like sudo -l to gather information. The exploit simply downloaded target files, but I had to guess at the names and locations. Not very helpful in the current scenario.



    Exploitation

    I moved onto trying the Cross-Site Request Forgery / PHP Code Execution (#40700) - which is where we begin our direct exploitation of the system, gathering both flags.

    This exploit was a bit more complex. It involved injecting PHP wrapped commands into the Ads section of the CMS dashboard, then returning to the directory /content/inc/ads, loading the ad file in-browser (double-click), then hard reloading the ads directory which resulted in a new Ad file containing your command response (unless the command runs silently).

    Below is an example of me running sudo -l where I revealed I can execute a specific script backup.pl with elevated privileges.

    Before moving on, it's important to note that our user.txt flag was found in the current working directory of www-data.

    Moving on to acquiring the root flag. I reviewed backup.pl and found that it was simply a proxy script executing /etc/copy.sh - this file was left with open rwx permissions.

    It was some kind of notification script, but that mattered little to me. Since I had write permissions I simply changed it to:

    find . root.txt && cat root.txt which immediately returned my root flag.

    Key Takeaways


    Version Enumeration via Changelog Always check for changelog.txt, readme, or version files during directory enumeration. They often reveal the exact software version and point you toward known exploits.


    Database Backups Are Gold SQL backup files are frequently left exposed. Always search for .sql, .bak, and backup files — they commonly contain plaintext credentials or password hashes.


    CMS Feature Abuse Once inside a CMS admin panel, explore every feature (especially content/ads/uploads). Many older CMS platforms allow direct PHP code injection through seemingly innocent fields.


    Sudo Script Analysis When you find a script you can run with sudo, always read it. Misconfigured scripts that call other files with weak permissions are a very common privilege escalation path.


    File Permission Enumeration World-writable files (especially ones executed by root via sudo or cron) are high-value targets. Always check permissions with ls -la on anything that looks important.
