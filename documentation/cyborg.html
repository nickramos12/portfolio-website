    TLDR

    Performed nmap reconnaissance on the target, enumerated directories with Gobuster, cracked an MD5 hash with Hashcat to find the password, extracted a BorgBackup archive to retrieve SSH credentials, logged in as user Alex, and escalated to root via a vulnerable sudo shell script that executed commands as root.

    Challenge Description

    Easy TryHackMe box involving encrypted archives (BorgBackup), source code analysis, hash cracking, and privilege escalation via a vulnerable sudo script.

    Reconnaissance

    I started with a full port scan to identify running services, and then ran the detected versions through searchsploit for any vulnerabilities.

    nmap -sV -F 10.66.178.64 - searchsploit apache 2.4.18 - searchsploit httpd 2.4 - searchsploit openssh 7.2p2

    I didn't find anything of immediate use, but I did come across a root privilege escalation exploit for Apache 2.4 (CVE-2019-0211) I found a working PoC and kept it on hand for later use.

    Click Here to view the exploit I found on Github.

    Since NMAP brought back an Apache web server, I attempted to pull it up in-browser. It loads in as "Apache2 Ubuntu Default Page" and the page's source code also provided no useful clues.

    From there, I decided to check for hidden directories with Gobuster. It came back with two directories and one webpage: /assets, /etc, and /index.html

    gobuster dir -u http://10.66.178.64 -w ~/Web-Content/common.txt -x php,html,js,txt,bak,old

    Navigating to /admin, I'm brought to what seems to be a musician's online portfolio.

    I dug through the website and found two clues: an admin chatlog and a compression file 'archive.tar'

    Alex admits his security controls are weak and doesn't know how to implement them correctly. He also hints at a backup called 'music_archive'

    Inside 'archive.tar' - I found a README file that describes the compression file as a BorgBackup Repo, with a link to the documentation.

    Navigating to /etc, I'm brought to a web directory that holds two files: passwd & squid.conf

    From the jump, the scrambled text resembled a hash digest (I came to this conclusion since some of the files in the BorgRepo indicating encryption) - so I pulled up Hashcat to experiment.



    First, I ran --identify to figure out the hashing algorithm, then I ran decryption with rockyou.txt and our digest came back as squidward.

    hashcat --identify hash.txt - hashcat -m 1600 -a 0 hash.txt ~/wordlists/rockyou.txt

    I would have to assume this is our BorgBackup repo passphrase. So I decided to read up on how to use BorgBackup.

    I found two critical commands: borg list and borg extract -- so I tried them out.

    After I extracted the backup repo, I ran find -name *.txt to quickly locate any txt files, and found Alex's password.

    Exploitation & Privilege Escalation

    Now that we have Alex's password, I connect via secure shell.

    I immediately found the user flag when checking the current directory: user.txt

    Ran sudo -l and discovered a custom script with a clear vulnerability. The script executed arbitrary commands passed via -c flag as root. I exploited the script to find the root flag.

    Key Takeaways


    Thorough Enumeration Wins Directory brute-forcing and manual browsing revealed critical files (archive + hash) that led to initial access.


    Hash Cracking Fundamentals Identifying hash type with hashcat and using quality wordlists is extremely effective against weak passwords.


    Backup Analysis Always inspect backup files/archives — they often contain sensitive data or credentials.


    Sudo Script Dangers Custom sudo scripts that execute user input without proper sanitization are a massive privilege escalation vector. Always review what you can run with sudo.
